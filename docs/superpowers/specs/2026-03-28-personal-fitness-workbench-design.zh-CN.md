# 个人健身工作台设计说明

日期：2026-03-28  
工作区：`personal-web` 根目录  
状态：已在对话中确认方向，当前版本用于中文审阅

## 1. 项目概述

这个网站是一个以中文为主的个人站点，包含两个核心部分：

- 一个公开的个人主页，用来展示你作为“产品 + 开发复合型”角色的定位
- 一个公开可用的健身工作台，任何人都可以直接使用
- `v1` 版本聚焦在“精细化每日代谢 / TDEE 计算器”

整体目标不是做一个普通作品集，也不是单纯做一个健身工具页，而是做成一个你自己平时会打开使用、同时对外也能清楚表达你是谁、做过什么、擅长什么的个人门户。

## 2. 已确认决策

- 核心定位：产品 + 开发复合型
- 主要受众：先满足自己日常使用，同时具备公开展示价值
- 公开边界：首页公开，工作台也公开并允许别人直接使用
- 主语言：中文
- 首页展示范围：定位、经历、项目、技能、联系方式
- `v1` 核心工具：每日代谢 / `TDEE` 计算器
- 数据存储：仅浏览器本地 `localStorage`
- 结果输出方式：既展示透明计算过程，也给简洁结论和建议
- 技术栈：`Vue 3 + JavaScript`
- UI 组件策略：允许使用 Element 体系；在 `Vue 3` 下对应为 `Element Plus`
- 环境约束：`v1` 默认不依赖额外安装新包

## 3. 产品目标

### 3.1 主要目标

- 让网站能清晰体现你具备产品判断力和动手实现能力
- 让网站成为你自己真正会用的日常入口，而不只是展示页
- 提供一个公开、可信、实用的健身热量计算工具
- 为后续扩展碳循环、碳水渐降、食物库、补剂库预留结构

### 3.2 `v1` 非目标

- 不做账号系统
- 不做后端和云同步
- 不在 `v1` 提供完整食物库浏览界面
- 不在 `v1` 提供完整补剂百科内容
- 不做复杂的身体重组规划系统
- 不提供医疗性质建议或个性化临床结论

## 4. 用户使用场景

### 4.1 你自己的使用场景

你会把这个站当成自己的个人入口页，打开后快速进入健身工作台，输入或修改体重、体脂、步数、训练量等信息，然后得到维持热量估算和实际可执行的饮食建议。

### 4.2 外部访问者的使用场景

访问者进入首页后，可以快速了解你的个人定位、经历、项目和技能；同时会注意到你做了一个真实可用的健身工作台，并且无需注册即可直接试用。

### 5.2 全局导航

- 首页

其中，“健身工作台”应该是导航里最强的入口按钮。

## 6. 首页设计

### 6.1 首页角色

首页承担两个任务：

- 用清晰、可信、有人味的方式介绍你自己
- 作为进入健身工作台的主要入口

### 6.2 首页模块

#### 首屏 Hero

- 中文主标题，明确传达“产品 + 开发复合型”身份
- 一段简短的个人定位说明
- 一个主要按钮，直接进入健身工作台
- 一个次级按钮，可跳转到项目或联系方式

#### 个人定位 / About

- 用更接近网页叙事的方式讲你的产品思维、开发执行力、自驱实验习惯
- 文案应更像经过设计的个人介绍，而不是直接照搬简历

#### 经历

- 基于 `assets` 里的简历内容整理成精简时间线或经历区块
- 强调角色、产出和影响，不照搬 PDF 简历版式

#### 项目

- 选择代表性项目，用偏结果导向的短描述呈现
- 当前这个健身工作台本身也应作为一个“正在成长中的项目”纳入

#### 技能

- 采用分组展示，而不是堆一整面关键词
- 可考虑的分组：产品、前端、工程实现、实验与数据、自我系统化

#### 联系方式

- 提供简洁可信的公开联系方式
- 风格轻量，不做复杂表单

## 7. 健身工作台设计

### 7.1 页面角色

`/fitness` 是一个公开、单页、可重复使用的工具页面。它应该像“工作台”，而不是像文章页。

### 7.2 页面主结构

#### 顶部说明区

- 用一句话说明这是一个面向训练人群的精细化 `TDEE` / 每日代谢估算工具
- 用简短提示说明：结果是估算值，不构成医疗建议

#### 输入面板

表单建议拆成几个清晰分组，避免变成长条表单：

- 身体信息：性别、年龄、身高、体重、体脂率
- 日常活动：日均步数、职业活动强度
- 训练信息：力量训练频率和时长、有氧训练频率和时长
- 目标语境：默认先给维持热量，同时展示减脂和增肌建议

#### 结果面板

结果需要同时包含两层：

- 透明计算层
- 直接建议层

透明计算层包含：

- 基础代谢估算
- 步数活动估算
- 职业活动修正
- 力量训练平均日消耗
- 有氧训练平均日消耗
- 最终 `TDEE`

直接建议层包含：

- 维持热量估算
- 减脂建议热量区间
- 精益增肌建议热量区间
- 对应场景下的三大营养素建议

#### 解释区

- 用中文解释结果是如何推导出来的
- 解释每个输入项代表什么
- 提醒用户应结合真实体重变化、训练表现和恢复情况持续修正

#### 后续模块入口卡片

`v1` 页面底部预留 4 张扩展入口卡片：

- 碳循环
- 碳水渐降
- 食物热量 / 宏量查询
- 补剂说明

这些入口在 `v1` 只作为结构预留，不阻塞首版上线。

## 8. `TDEE` 计算模型

### 8.1 设计原则

这个工具应该比普通“一键活动系数”计算器更透明，但又不能假装自己能做到绝对精准。

### 8.2 基础代谢计算

建议逻辑如下：

- 如果用户填写了体脂率，优先使用更适合瘦体重估算的公式，例如 `Katch-McArdle`
- 如果没有体脂率，则回退到更通用的标准公式，例如 `Mifflin-St Jeor`

这样既能利用更细的数据，又不会让普通访问者因为缺少体脂率而无法使用。

### 8.3 活动与训练消耗模型

建议拆分为以下部分：

- 步数用于估算日常移动带来的能量消耗
- 职业活动强度用于修正日常非运动活动水平
- 力量训练与有氧训练基于每周总量折算成平均每日能量贡献

界面上应明确标注：这些都是估算项，而不是绝对值。

### 8.4 输出场景

计算器应始终给出 3 种热量场景：

- 维持
- 减脂
- 精益增肌

宏量营养素分配建议：

- 优先锚定蛋白质
- 脂肪设置最低保护下限
- 剩余热量由碳水补足

这样做可以让后续扩展碳循环、碳水渐降时，直接复用核心热量引擎。

## 9. 本地数据与内容组织

### 9.1 浏览器本地存储

所有工具输入与最近结果记录都使用 `localStorage` 保存。

建议保存内容：

- 最近一次表单输入值
- 最近几次计算结果历史
- 如果后续加入单位切换或显示偏好，也可以一起保存

### 9.2 基于简历的个人内容来源

首页内容不直接渲染 PDF 简历，而是把简历内容整理成网站可维护的数据结构。

建议内容来源：

- `assets/` 下的开发向简历 PDF
- `assets/` 下的产品向简历 PDF

建议处理方式：

- 手动整理成结构化本地数据文件
- 网站文案应更精炼、更网页化
- 不复刻 PDF 简历排版

## 10. 视觉方向

### 10.1 整体基调

整个站点应同时具备两种气质：

- 首页偏“编辑感 / 个人品牌感”
- 工作台偏“仪表板 / 工具感”

它不应该长得像：

- 通用程序员作品集模板
- 健身 App 的网页翻版
- 纯表格式、堆数字的计算器

### 10.2 美术与排版方向

建议方向：

- 重视中文排版体验和标题层级对比
- 首页留白更大胆，布局可带一点非对称和编辑感
- 工作台页面强调数字结果、卡片层级和表单清晰度
- 两个页面共用一套主题颜色和视觉语言，保证仍像一个整体产品

### 10.3 交互风格

- 首页交互可以更有展示感和节奏感
- 工作台交互要稳定、直接、高效
- 表单变化后的结果反馈要明显，但不要太花哨

## 11. 技术架构

### 11.1 `v1` 交付策略

由于本项目明确要求“尽量不额外安装新依赖”，因此 `v1` 默认采用“零安装优先”策略。

建议实现方式：

- 使用浏览器运行时的 `Vue 3`
- 全部采用 `JavaScript`
- 使用本地 ES Modules 组织代码
- 路由如有需要，优先采用轻量可直接运行的方式
- 如果使用 `Element Plus`，也要以不增加本地安装负担为前提

这意味着首版更适合采用“静态页面 + 本地模块化脚本”的方式，而不是一开始就依赖完整构建链路。

### 11.2 Element 体系说明

你提到“也可以使用 Element UI”。在 `Vue 3` 语境下，对应的兼容选择应为 `Element Plus`。

因此本项目的解释是：

- 允许使用 `Element Plus` 来优化表单、卡片、布局体验
- 如果在“零安装优先”的前提下集成成本偏高，则回退为自定义 `Vue 3` 组件 + 项目内 CSS

### 11.4 职责划分

- 个人信息和简历整理内容放在 `data/`

即使不依赖完整构建工具，这种结构也能保持代码清晰、方便后续扩展。

## 12. 风险与应对

### 12.1 风险：结果看起来过于精准

如果数字表达太“绝对”，用户可能会过度相信结果。

应对方式：

- 明确标注“估算值”
- 展示计算结构而不是只给单一结果
- 对减脂和增肌给区间建议，而不是单点值

### 12.2 风险：首页和工具页像两个网站

如果首页太偏个人展示、工作台太偏工具，整体会割裂。

应对方式：

- 使用同一套视觉系统
- 在两个页面中复用相同的个人定位语言
- 把健身工作台当成你的公开身份组成部分，而不是隐藏附属页

### 12.3 风险：零安装约束降低开发便利性

在不依赖构建链路的情况下，不能天然使用 `.vue` 单文件组件，也不适合直接上很多包管理依赖。

应对方式：

- 从一开始就把代码组织成清晰的本地模块
- 保持组件边界明确
- 为未来迁移到 `Vite` 保留空间；如果以后你愿意安装依赖，可以平滑升级
