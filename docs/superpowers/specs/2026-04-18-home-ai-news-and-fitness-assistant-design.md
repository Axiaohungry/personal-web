# 首页 AI 最新动态与健身助手设计说明

日期：2026-04-18  
工作区：`personal-web` 根目录  
状态：已在对话中确认方向，当前版本用于进入实现计划前的审阅

## 1. 改动目标

本次设计说明只处理两个增量模块：

- 首页首屏附近新增一个醒目的 `AI 最新动态` 信息模块
- 健身工作台新增一个严格限定范围的 `训练与健康助手`

本轮不是首页重构，也不是把健身工作台改造成通用聊天产品。目标是在不打乱现有信息结构的前提下，补上两个高价值入口：

- 对招聘方：更快看到作者对 AI 工具与生态变化的持续关注，以及把实时信息整理成产品化界面的能力
- 对普通用户：在现有健身工作台里，获得更直接的训练、饮食、恢复与补剂建议入口

## 2. 已确认方向

- 首页只做增量增强，不重排现有主体结构
- `AI 最新动态` 必须出现在刚进入主页后较醒目的位置
- 首页 `AI 最新动态` 不放作者个人短判断，不做评论栏
- 新闻内容继续使用 Gemini 路线，但必须保证时效性和来源可追溯
- 健身页新增 `训练与健康助手`，位置在当前核心计算与模块入口之间
- `训练与健康助手` 只回答训练与健康相关问题，其他问题不回答
- 助手边界不包含医疗诊断、处方建议、化验解读和非健身领域问答

## 3. 总体策略

本次新增遵循：

`强入口，轻改动，重边界`

具体含义如下：

- 强入口：模块必须足够明显，让用户进入页面后能快速注意到
- 轻改动：不推翻首页 hero、健身工作台主体和现有模块容器
- 重边界：新闻模块强调来源与刷新逻辑，助手模块强调范围限制与拒答机制

## 4. 首页模块设计

### 4.1 位置

`AI 最新动态` 放在首页 hero 区块之后、正文主体之前，作为一块独立的 `shell-surface` 模块插入 [src/views/HomeView.vue](</G:\Codex—playground\personal-web\src\views\HomeView.vue:1>)。

推荐顺序：

1. `SiteHeader`
2. `HomeHero`
3. `AI 最新动态`
4. `home-overview`
5. 其余现有内容区块

这个位置可以保证：

- 进入首页后很快被看到
- 不破坏现有 hero 结构
- 不把重要动态埋进较深正文

### 4.2 视觉形态

模块定位为：

`编辑感的情报带，而不是新闻门户区`

视觉要求：

- 延续现有暖米色、赤陶色、墨绿色体系
- 信息层级依靠标题、时间、分隔和卡片节奏体现，不用高饱和抢眼色
- 保持轻微进入动效，但不做轮播、不做自动切换
- 浅色和深色主题都应维持同一套克制、成熟的气质

### 4.3 模块结构

左侧信息区：

- 标题：`AI 最新动态`
- 更新时间：例如 `更新于 2026 年 4 月 18 日 21:30`
- 简短功能说明：例如“聚合最近值得关注的 AI 进展，优先保留对产品、工作流和工具生态有直接影响的变化。”
- 操作按钮：
  - `查看来源`
  - `刷新动态`

右侧事件区：

- 默认展示 3 条事件卡
- 每条卡片字段固定：
  - `title`
  - `summary`
  - `whyItMatters`
  - `sourceLabel`
  - `publishedAt`

明确不展示：

- 作者个人评论
- 大段正文
- 自动轮播
- 多层嵌套标签系统

### 4.4 移动端要求

- 模块改为上下堆叠
- 顶部先显示标题、更新时间和说明
- 事件卡垂直排列
- 首屏至少应完整露出 1 条事件卡，而不是压缩成摘要墙

## 5. 首页新闻的数据与刷新策略

### 5.1 接口

新增同源接口：

- `GET /api/ai/news-brief`

### 5.2 返回结构

```json
{
  "updatedAt": "2026-04-18T21:30:00+08:00",
  "stories": [
    {
      "id": "story-1",
      "title": "标题",
      "summary": "一句话说明发生了什么",
      "whyItMatters": "为什么值得关注",
      "sourceLabel": "Google AI Blog",
      "sourceUrl": "https://example.com",
      "publishedAt": "2026-04-18"
    }
  ],
  "sourceIndexUrl": "https://example.com"
}
```

### 5.3 生成规则

- 仅返回 3 条事件
- `summary` 控制为 1 句
- `whyItMatters` 控制为 1 句
- 每条事件必须带来源链接
- 选题优先级偏向：
  - 模型能力边界变化
  - 产品工作流影响
  - 开发者工具与生态变化
  - 重要平台发布
- 不追求泛娱乐式热点

### 5.4 技术策略

首页新闻继续走 Gemini 路线，但必须升级为：

`Gemini + Google Search grounding + structured JSON + server-side cache`

原因如下：

- 当前 `server/fitnessGemini.js` 的裸生成模式适合食物库与补剂库这类弱时效检索
- 首页 `最新 AI 大事` 是强时效场景，不能只依赖模型静态知识
- 官方文档已经支持 Gemini API 的 Google Search grounding 和 structured outputs，因此可以在同一条服务端路线内完成实时搜索、来源引用与 JSON 归一

### 5.5 缓存与失败策略

- 服务端缓存：`15-30 分钟`
- 缓存命中时直接返回，降低首页首屏延迟和调用成本
- 接口失败时，前端模块仍然保留，只显示降级状态：
  - `暂时无法刷新最新动态，请稍后再试`
- 不允许因为接口失败导致首页布局塌陷或整块消失

## 6. 健身页助手设计

### 6.1 位置

`训练与健康助手` 放在 [src/views/FitnessView.vue](</G:\Codex—playground\personal-web\src\views\FitnessView.vue:1>) 中：

- `TdeeSummary` 之后
- `FutureModules` 之前

这个位置可以利用用户刚输入完的上下文信息，并把助手作为“接下来怎么做”的桥梁，而不是独立聊天页。

### 6.2 定位

该模块是：

`上下文敏感的训练与健康建议面板`

不是：

- 通用 AI 聊天框
- 医疗问诊入口
- 健身之外的知识助理

### 6.3 视觉与交互形态

助手采用与现有 `fitness-panel` 一致的面板化表达，不做悬浮球或聊天浮窗。

模块结构：

上部：

- 标题：`训练与健康助手`
- 范围说明：只回答训练、饮食、恢复、补剂和一般健康习惯相关问题
- 当前上下文摘要：
  - 当前目标
  - 周期
  - 参考热量

中部：

- 文本输入框
- 提交按钮
- 4 到 6 个快捷问题 chips

下部：

- 结构化回答卡
- 错误卡
- 拒答卡

### 6.4 快捷问题方向

默认快捷问题示例：

- `减脂期训练前后怎么吃更稳？`
- `我这周恢复差，要不要降训练量？`
- `按我现在的数据，蛋白应该怎么分配？`
- `肌酸和咖啡因适不适合当前目标？`
- `增肌期步数和有氧怎么安排更合适？`

## 7. 助手回答范围与拒答边界

### 7.1 允许回答的范围

助手只回答以下五类问题：

- 训练安排：频率、分化、容量、强度、恢复节奏
- 饮食执行：热量、蛋白、三大营养素、进餐时机、减脂与增肌策略
- 恢复习惯：睡眠、疲劳、训练后的恢复建议
- 补剂判断：是否值得用、常见剂量、适用场景、注意事项
- 一般健康习惯：饮水、步数、久坐、作息等基础生活方式建议

### 7.2 明确不回答的范围

- 与健身或健康无关的问题
- 泛知识、新闻、情感、职场、编程等其他领域问题
- 医疗诊断、疾病判断、化验单解释、处方药建议
- 极端减重、危险饮食和伤害性内容

### 7.3 拒答策略

超出范围时，不尝试“顺便回答一点”，而是直接返回结构化拒答结果。

拒答核心语义：

- 该助手只处理训练、饮食、恢复、补剂和一般健康习惯相关问题
- 引导用户改问一个合规问题

医疗边界时必须单独返回：

- 该问题已接近医疗判断边界
- 不能代替医生进行诊断或治疗建议

## 9. 助手边界控制实现

边界控制不能只依赖 prompt，必须至少做三层：

### 9.1 第一层：请求前置判定

服务端先判断问题是否属于以下域：

- 训练
- 饮食
- 恢复
- 补剂
- 一般健康习惯

明显不相关的请求，直接返回 `out_of_scope`。

### 9.2 第二层：Gemini 系统指令

系统指令必须明确：

- 只能回答训练与健康相关问题
- 禁止输出非相关领域内容
- 禁止医疗诊断、处方和化验解释
- 必须输出固定 JSON

### 9.3 第三层：响应后校验

服务端必须验证：

- 返回是否为合法 JSON
- 字段是否齐全
- 状态是否在允许集合内
- 文本内容是否越过医疗边界或通用问答边界

如果模型跑偏，直接降级为固定拒答结构，而不是把失控内容传给前端。

## 10. 与现有模块的衔接

`relatedModules` 不应完全交给模型自由生成，而应由本地规则映射：

- 食材与餐食选择 -> `食物库`
- 补剂问题 -> `补剂库`
- 热量与目标策略 -> `增肌底层热量逻辑`
- 训练安排与分部位计划 -> `谭成义焚诀训练体系`

这样可以让助手像“工作台中的引导层”，而不是孤立的回答器。

## 12. 验收标准

完成后至少应满足以下结果：

- 用户进入首页后，不滚动太久就能看到 `AI 最新动态`
- 新闻模块在视觉上足够醒目，但不破坏现有 hero 和正文节奏
- 首页新闻始终附带来源，且失败时有稳定降级状态
- 健身工作台中的助手在输入上下文后可给出结构化建议
- 助手只回答训练与健康相关问题，超范围时稳定拒答
- 助手不会滑向通用聊天框
- 桌面端与移动端都保持完整可用体验

## 13. 风险与缓解

### 13.1 新闻时效与准确性风险

风险：

- 裸生成会产生过时或不可信内容

缓解：

- 必须使用 grounding
- 必须保留来源链接
- 必须做缓存与失败降级

### 13.2 助手越界风险

风险：

- 问题被包装成健康相关，但实质是医疗或无关领域

缓解：

- 前置判定
- 系统指令
- 响应后二次校验

### 13.3 模型输出不稳定风险

风险：

- JSON 不稳定或字段缺失

缓解：

- 使用 structured output
- 服务端 schema 校验
- 失败时回退到固定错误结构
