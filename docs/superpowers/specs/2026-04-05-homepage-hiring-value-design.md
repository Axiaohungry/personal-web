# 首页招聘价值重排设计说明

日期：2026-04-05  
工作区：`personal-web` 根目录  
状态：已在对话中确认方向，当前版本用于进入首页改造实现前的审阅

## 1. 改版目标

本次设计说明只面向首页与其直接关联的招聘表达问题，不处理健身工具首用成本的完整重构。

当前首页已经具备明确的个人气质、完整的案例内容和可用的视觉基底，但仍存在以下问题：

- 首屏表达了“我怎么想”，却没有足够快地证明“我做过什么”
- 招聘方进入首页后，需要滚动到较后位置才看到最强业务案例
- 首页当前默认第一案例是自用工具，而不是最强的业务证明
- 个别案例页将公开版占位材料直接命名为“证据”，并暴露“后续替换真实素材”这类制作中语气
- 局部中英混用和字体外链策略，会削弱“理智、成熟、稳定”的品牌感

本次改版的目标不是推翻现有首页，而是把它从“表达清楚自己的判断”提升为“先建立招聘信任，再展开个人方法论”。

## 2. 已确认方向

- 首页策略：`保留判断力表达，但改成招聘摘要版首屏`
- 招聘默认角色定位：`解决方案 / 产品 / 前端`
- 首页默认第一主案例：`政务审批地图项目`
- 首屏路径：保留双入口，但不做重分流首页
- 语义策略：公开版材料不再冒充“证据”，而是明确为公开说明或结构示意
- 本轮范围：优先修复首页招聘价值表达，不同步展开健身工具首用成本优化

## 3. 总体设计策略

本次首页采用：

`先证明，再解释`

具体来说：

- 首屏先让招聘方知道你是做什么的人
- 首屏后立刻看到最强业务案例，而不是先读个人叙述
- 方法论与个人表达继续保留，但下沉到业务证明之后
- 公开版案例素材统一切换到更诚实、更成熟的招聘语义

这个策略不会抹掉你原有的理智、克制、编辑感气质，只是调整信息顺序，让招聘方在前 10 秒先建立“可雇佣价值”的判断。

## 4. 首页结构改版

### 4.1 新的首页顺序

首页主结构调整为：

1. `招聘摘要版首屏`
2. `代表案例`
3. `方法补充 / 在做的事`
4. `能力与工具`
5. `联系`

核心变化是把原来的 `projects` 区整体前提，放在 `experience` 和叙事型说明之前。

### 4.2 首屏职责

首屏只负责回答 3 个问题：

1. 你是谁  
2. 你主要解决什么问题  
3. 招聘方下一步应该看哪里

首屏不再承担完整自我介绍，也不再堆叠过多“表达型信息”。

### 4.3 方法型内容的去中心化

当前首页里的 `narrativeCards` 不是问题本身，问题在于它们出现得太早。

改版后：

- 这些内容继续保留
- 但位置下沉到代表案例之后
- 职责从“首屏叙事核心”改为“看完案例后的方法补充”

这样既保住你的思考力表达，也不会让它盖过业务证明。

## 5. 首屏内容设计

### 5.1 角色表达

首页首屏统一使用：

`解决方案 / 产品 / 前端`

这组角色顺序比当前“产品 + 前端”更准确，也更符合你希望招聘方记住的能力结构。

它传达的不是“会做产品的前端”，而是：

- 能理解业务与约束
- 能组织方案
- 能推进到页面与交付层

### 5.2 主标题策略

主标题继续保留你的判断力气质，但改成更像招聘陈述，而不是纯作者表达。

目标语义应当是：

`我做解决方案，也做产品和前端，擅长把复杂业务整理成可交付、可使用、可迭代的体验。`

允许后续在实现时微调措辞，但必须保留 4 个关键信息：

- 解决方案
- 产品
- 前端
- 复杂业务落地

### 5.3 正文信息减负

当前 `HomeHero.vue` 中的大标题、tagline、summary、focus、signals、stats 同时出现，导致招聘方需要先解析作者表达，才能找到业务证明。

改版后，首屏只保留：

- 一句主标题
- 一句事实型招聘摘要
- 两个 CTA
- 一组轻量的事实摘要
- 一个右侧招聘速览卡

以下内容不再以高权重方式同时出现在首屏正文里：

- 长 summary
- `/` 连接的 focus line
- 偏做事习惯和长期兴趣的 signal 卡片
- 双卡叠加式的编辑感 aside

### 5.4 CTA 设计

首屏保留两个明确行动入口：

- 主按钮：`先看代表案例`
- 次按钮：`打开健身工具`

这样可以保留双路径，但优先级更明确：

- 招聘方默认去案例
- 普通用户仍可直接进入工具

### 5.5 右侧速览卡

右侧 aside 从“表达质感卡组”改成“招聘速览卡”。

速览卡优先展示：

- 角色：`解决方案 / 产品 / 前端`
- 代表场景：`政务审批地图 / 工具型产品`
- 默认入口：`先看代表案例`

它的职责是帮助招聘方快速扫描，而不是继续铺陈个人表达。

## 6. 代表案例区改版

### 6.1 政务审批地图固定为第一主案例

首页案例区采用：

`1 个主案例 + 2 到 3 个补充案例`

其中 `approval-map-workflow` 固定为首页第一主案例。

这是本次首页改造最关键的结构变化，因为它最能证明你在真实业务约束下的能力闭环：

- 解决方案推进
- 产品结构整理
- 前端协同落地
- 复杂流程收口为可交付页面

### 6.2 主案例卡的写法

首页主案例卡不再只是作品介绍，而是招聘摘要卡。

它需要优先回答：

1. 业务场景是什么  
2. 你负责什么  
3. 难点是什么  
4. 最后证明了什么

首页层面的文案应该优先使用：

- 场景
- 角色
- 约束
- 结果

而不是优先使用：

- 个人感受
- 长叙事
- 近似指标

### 6.3 补充案例排序

推荐首页案例排序为：

1. `政务审批地图项目`
2. `3DGS 三维热场重建研究`
3. `校园产品化协作项目集`
4. `健身工具 / 健身服务相关案例`

排序逻辑如下：

- 第一位证明真实业务复杂度
- 第二位证明陌生问题学习与抽象能力
- 第三位证明组织推进与协作能力
- 健身相关承担长期工具化与真实使用补充

### 6.4 案例区标题语气

当前“几个能代表我的切面”偏作者视角。

改版后应更接近招聘阅读语境，例如：

- 标题：`代表案例`
- 引导语：`先看我在真实业务、复杂问题和长期工具中的几类代表性工作。`

语气要继续克制，但要更成熟、更职业。

## 7. 公开版材料与可信度修词

### 7.1 不再滥用“证据”

当前 `approvalMapWorkflow.js` 与 `fitnessCoaching.js` 中的公开版占位材料，仍然使用“证据”框架，并直接显示“后续替换真实素材”“匿名安全占位图”等说明。

这在招聘语境里会被解读成：

- 案例还没完成
- 当前页仍在占位阶段
- 材料还不够可靠

改版后应采用更清晰的二分策略：

- 有真实匿名截图时，允许使用 `证据`
- 仍是公开版材料时，统一改成 `公开版说明`、`结构示意` 或 `公开材料片段`

当前轮次优先采用第二种。

### 7.2 公开版材料的职责

公开版材料的目标不是证明“我确实有很多东西”，而是帮助招聘方理解：

- 复杂信息是怎么被整理的
- 流程是怎么被收束成页面结构的
- 哪些约束被显性化处理了

因此，这些材料应当被定义为：

`结构说明型内容`

而不是：

`战果证据型内容`

### 7.3 hero rail 的内容修正

当前个别案例 hero 中使用了近似比例和近似效率提升数字。

在没有严格可核验材料的情况下，这类数字会带来额外追问，并削弱页面可信度。

改版后，案例页 hero rail 优先呈现：

- 角色
- 职责
- 约束
- 结果属性

例如，比起“审批步骤 -30%（近似）”，更可信的表达是：

- `多角色审批流被收束到同一条阅读路径`
- `地图对象、表单与结果回写被组织进统一页面结构`

### 7.4 占位说明的语气

需要明确避免继续使用这类制作中语气：

- `后续替换真实素材`
- `临时安全占位图`
- `之后可替换成真实截图`

替代写法应当更像公开展示边界说明，例如：

- `因公开展示限制，此处保留匿名裁切后的结构示意`
- `以下材料用于说明页面结构与流程组织方式`
- `公开版仅展示不涉敏感信息的界面片段`

这些写法仍然诚实，但不会制造“未完成”感。

## 8. 微文案与基础设施修正

### 8.1 文案统一

首页和工具页中存在局部中英混用，例如：

- `Profile`
- `Inputs`
- `Body`
- `Activity`
- `Navigate`

这会轻微削弱品牌的一致性和成熟感。

本轮首页改造应至少覆盖以下层：

- `HomeHero.vue`
- `SiteHeader.vue`
- 首页关键按钮与标签
- 直接与首页招聘叙事相连的案例区 copy

原则是：

- 招聘主路径统一中文
- 仅在技术名词不可替代时保留英文

### 8.2 字体策略

`tokens.css` 当前直接通过 Google Fonts 外链加载字体。

这对中国网络环境和首屏稳定性存在隐性风险，也不符合“成熟、可控”的基础设施判断。

本轮的最小修复目标是：

- 去掉对 Google Fonts 的直接依赖
- 使用更稳的本地字体栈或项目内可控字体方案

本轮不强制一次性完成完整 self-host 字体资产建设，但必须先消除首页对外部字体网络的直接依赖。

### 9.1 本轮必改文件

- `src/components/HomeHero.vue`
- `src/views/HomeView.vue`
- `src/data/projects.js`
- `src/data/projectCases/approvalMapWorkflow.js`
- `src/data/projectCases/fitnessCoaching.js`
- `src/components/projects/ProjectEvidenceGrid.vue`
- `src/components/SiteHeader.vue`
- `src/styles/tokens.css`

### 9.2 本轮只做轻微对齐的文件

- `src/data/experience.js`
- `src/data/contact.js`
- `src/data/skills.js`

这些文件如需配合首页语气做轻微文案对齐可以进行，但不应在本轮被重写为新的叙事系统。

### 9.3 本轮明确不展开的范围

以下问题已被确认存在，但不纳入本轮实现主目标：

- `FitnessView.vue` 的首用路径重构
- `TdeeForm.vue` 的基础模式 / 进阶模式
- `TdeeSummary.vue` 的默认推荐方案强化
- `FutureModules.vue` 的 iframe 替换为页内路由或自适应模块容器
- 全量真实匿名素材补齐

这些应在后续“健身工具首用成本优化”专题中单独设计。

## 10. 验收标准

本轮完成后，首页至少应满足以下结果：

- 招聘方在 10 秒内能知道你是做 `解决方案 / 产品 / 前端` 的
- 首页首屏后立刻看到最强业务案例，而不是先进入个人表达区
- `approval-map-workflow` 成为默认第一主案例
- 公开版案例材料不再因“证据”命名和占位语气伤害招聘信任
- 首页与导航关键微文案不再出现明显中英混用
- 去掉 Google Fonts 外链后，首屏排版仍稳定成立
- 桌面端与移动端都保留清晰、克制、成熟的阅读体验

## 11. 审阅备注

- 当前方向已在对话中获得用户确认
- 下一步不应继续发散讨论视觉路线，而应在用户审阅本 spec 后直接进入实现计划
- 后续实现计划应围绕“首页招聘价值提升”收口，不把健身工具体验重构混入同一轮开发
