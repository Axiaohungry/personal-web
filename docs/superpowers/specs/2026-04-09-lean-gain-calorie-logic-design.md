# 增肌底层热量逻辑设计说明

日期：2026-04-09  
工作区：`personal-web` 根目录  
状态：已在对话中确认方向，当前版本用于进入实现计划前的审阅

## 1. 模块目标

本次新增模块名为：`增肌底层热量逻辑`

它的职责不是重复做一次主页面已有的 `TDEE` 计算，而是在主页面已经得到基础代谢、总消耗和体脂率之后，继续回答 5 个更接近实际使用的问题：

- 我当前处在增肌逻辑的第几阶
- 现在更适合热量赤字、持平、还是盈余
- 每天蛋白质、脂肪、碳水应该怎么分
- 本周是否需要根据体重变化下调碳水
- 在任何情况下，哪些安全边界不能被突破

这个模块应继续沿用现有健身工作台的结构：

- 在 `/fitness` 的功能模块区出现一个新 tab
- 点击后通过 `iframe` 嵌入独立模块页
- 模块页接收主页面透传的上下文
- 模块结果可被本地持续记录和回看

## 2. 已确认设计边界

### 2.1 直接复用主页面结果

模块不重复计算以下字段，而是直接复用主页面已有结果：

- `Gender`
- `Age`
- `Height`
- `Weight`
- `BodyFat_Pct`
- 主页面已计算出的 `BMR`
- 主页面已计算出的 `TDEE`

### 2.2 本次不做的内容

为了保持模块清晰、可落地且不引入冗余参数，本次明确不做：

- 不单独采集 `Activity_Level`
- 不在模块内重复计算 `BMR / TDEE`
- 不做每日体重记录
- 不做 `7-day SMA`
- 不做腰围、围度、照片上传
- 不做训练日 / 非训练日差异化碳循环
- 不做后端数据库与定时任务

### 2.3 本次新增的最小输入

模块只新增两个必要输入：

- `Exp_Level`：训练经验，取值为 `Novice / Advanced`
- `Weekly_Avg_Weight`：用户手动填写的本周平均体重

这两个输入足以支撑“主动盈余期”和“重组期”的周度调控判断，同时不会把模块做成新的记录中心。

### 2.4 体脂率是硬前置条件

这个模块的核心判断变量是 `BodyFat_Pct`。

因此模块在进入正式计算前，必须先做前置校验：

- 如果主页面体脂率为空
- 或为 `0`
- 或小于等于 `0`
- 或不是有效数字

则模块不进入：

- 阶段判定
- 目标热量计算
- 宏量营养素分配
- 周度调控判断

而是直接展示提醒态。

提醒态的表达应是产品化引导，而不是报错：

- 标题：`先补充体脂率，再看这套逻辑`
- 说明：`这个模块的判断核心不是体重本身，而是你现在处在哪个体脂阶段。先回到主页面补上体脂率，这里才能给出更可靠的热量和宏量建议。`

这样可以避免用户在关键输入缺失时得到误导性的结果，也能更清楚地体现产品的判断边界。

## 3. 产品策略

这个模块应该呈现为：

`解释底层逻辑 + 给出目标热量 + 做周度调控`

而不是：

`另一个热量计算器`

它的产品价值主要体现在两个层面：

- 对普通用户：把“为什么这样吃”讲明白，让建议更容易执行
- 对招聘方：展示你如何把一套偏教练 / 算法化的逻辑整理成产品结构、判断层和连续使用体验

## 4. 核心算法设计

### 4.1 四阶分段逻辑

阶段判定完全基于体脂率，并按性别区分阈值。

#### 男性

- 第一阶：`BodyFat_Pct >= 20%`
- 第二阶：`15% <= BodyFat_Pct < 20%`
- 第三阶：`12% <= BodyFat_Pct < 15%`
- 第四阶：`BodyFat_Pct < 12%`

#### 女性

- 第一阶：`BodyFat_Pct >= 30%`
- 第二阶：`25% <= BodyFat_Pct < 30%`
- 第三阶：`20% <= BodyFat_Pct < 25%`
- 第四阶：`BodyFat_Pct < 20%`

模块需要输出：

- `phase`
- `phaseLabel`
- `phaseStrategy`
- 适合的热量方向说明

### 4.2 目标热量算法

模块直接基于主页面传入的 `TDEE` 计算 `Target_Calories`。

#### 第一阶：高体脂重组期

- 策略：热量赤字
- 公式：`Target_Calories = TDEE - 300`

#### 第二阶：平衡重组期

- 策略：热量持平
- 公式：`Target_Calories = TDEE`

#### 第三阶：黄金增肌期

- 策略：微弱盈余
- 公式：`Target_Calories = TDEE * 1.065`

#### 第四阶：主动盈余期

- 策略：主动盈余
- 公式：`Target_Calories = TDEE * 1.10`

这里不开放 `+8% ~ +12%` 的可调参数。  
第一版默认采用中位策略，以降低参数噪音。页面文案中明确说明：极难增肌者后续可以再上调，但当前模块默认按中位数执行。

### 4.3 宏量营养素算法

模块严格采用：

`先定蛋白质，再定脂肪，最后由碳水补齐`

#### 蛋白质

- 固定系数：`1.6 g/kg`
- 公式：`Protein_g = Weight × 1.6`
- 热量换算：`Protein_Kcal = Protein_g × 4`

第一版不开放蛋白因子调节，以避免冗余参数。

#### 脂肪

脂肪因子按体脂阶层与性别动态决定。

##### 男性

- 第一阶：`0.6 g/kg`
- 第二阶：`0.7 g/kg`
- 第三阶：`0.8 g/kg`
- 第四阶：`1.0 g/kg`

##### 女性

- 第一阶：`0.7 g/kg`
- 第二阶：`0.8 g/kg`
- 第三阶：`0.8 g/kg`
- 第四阶：`1.0 g/kg`
- 特例：若女性体脂 `< 15%`，脂肪因子提高至 `1.2 g/kg`

公式：

- `Fat_g = Weight × Fat_Factor`
- `Fat_Kcal = Fat_g × 9`

#### 碳水

- `Carb_g = (Target_Calories - Protein_Kcal - Fat_Kcal) / 4`

如果进入周度调控，惩罚只作用于碳水，不会下调蛋白和脂肪。

## 5. 周度调控引擎

### 5.1 设计原则

由于本次不采集每日体重、围度和照片，周度调控逻辑需要从视频里的“完整观测模型”简化为“前端轻量可用模型”。

本模块采用：

- 用户手动录入 `Weekly_Avg_Weight`
- 本地保存最近若干周记录
- 每次录入后即时判断是否需要下一周碳水 `-10%`

### 5.2 第一、二阶逻辑

第一、二阶对应：

- 高体脂重组期
- 平衡重组期

简化判定规则：

- 若历史记录不足 3 周：不触发惩罚，只提示继续观察
- 若最近 3 周 `Weekly_Avg_Weight` 连续上升：判定当前热量可能偏高，下一周碳水 `-10%`

这样做的代价是判断会比“体重 + 腰围 + 视觉”更粗，但与当前“轻输入、前端可用”的模块定位一致。

### 5.3 第三、四阶逻辑

第三、四阶对应：

- 黄金增肌期
- 主动盈余期

判定方式：

- `Weight_Delta_Pct = (Current_Weekly_Avg_Weight - Last_Weekly_Avg_Weight) / Last_Weekly_Avg_Weight`

阈值：

- 新手：`Weight_Delta_Pct > 0.5%`
- 老手：`Weight_Delta_Pct > 0.25%`

若超过阈值：

- 判定为盈余过大
- 下一周碳水 `-10%`

### 5.4 调控结果输出

模块需要清楚输出：

- `adjustmentDecision`
- `adjustmentReason`
- `carbAdjustmentPct`
- 调整后的 `Carb_g`

并把文案写成人话，例如：

- `本周先维持不动`
- `本周建议下周碳水 -10%`
- `历史周数不足，先继续观察`

## 6. 安全围栏

无论热量策略还是碳水惩罚如何变化，最终结果都必须经过以下围栏：

- 女性总热量不低于 `1200 kcal`
- 男性总热量不低于 `1500 kcal`
- 脂肪不低于 `0.5 g/kg`
- 惩罚只减碳水，不减蛋白和脂肪

模块页中应明确展示这些保护条件，避免把结果表现成“绝对算法答案”。

## 7. 页面结构设计

新模块页继续使用 `ModuleWorkbenchLayout`，整体结构分为 6 个信息区。

若体脂率缺失，则不显示以下 6 个正式信息区，直接切换为提醒态。

### 7.1 顶部总览

这一块用来快速回答：

- 你在第几阶
- 当前建议热量是多少
- 本周是否触发调控

适合用统计卡 + 状态提示的组合。

### 7.2 底层逻辑解释

通过可展开的解释区回答：

- 为什么你会落在这个阶段
- 为什么热量这样设
- 为什么宏量营养素这样分

这部分承担“把复杂逻辑讲清楚”的职责。

### 7.3 本周输入区

只出现两个用户输入：

- 训练经验
- 本周平均体重

不出现额外技术参数。

### 7.4 周度调控结果区

单独展示：

- 当前判断
- 触发原因
- 调整后的碳水目标

让用户一眼看懂“这周需要做什么变化”。

### 7.5 历史记录区

展示最近几周本地记录：

- 周均体重
- 所处阶段
- 是否触发碳水下调
- 当周碳水目标

这块的作用是把模块从“一次性答案”变成“连续使用工具”。

### 7.6 提醒与边界区

集中展示：

- 本模块建立在主页面 `TDEE` 已较合理的前提上
- 女性低体脂阶段的脂肪底线会自动抬高
- 所有建议都受热量和脂肪围栏保护
- 这是营养策略起点，不替代医疗判断

## 8. 本地存储设计

为了不和主页面现有 `latest / history` 结构混在一起，本模块使用独立存储键。

### 8.1 偏好存储

`leanGainLogicPrefs`

包含：

- `expLevel`
- `lastWeeklyAverageWeight`

### 8.2 历史存储

`leanGainLogicHistory`

每条记录包含：

- `savedAt`
- `weeklyAverageWeight`
- `phase`
- `targetCalories`
- `proteinGrams`
- `fatGrams`
- `carbGrams`
- `carbAdjustmentPct`
- `adjustmentDecision`
- `adjustmentReason`

该结构应控制数量上限，避免本地存储无限增长。
